1同期タイミング: 常時同期モデル

回答 各種編集アクション（変数作成・カテゴリ作成・移動・編集保存・削除）で同期させたい。常に同期状態を維持。

✓ 決定: アクションフック型・全自動同期 + debounce

Manager 内のすべての書き込み系操作で state.save() が呼ばれている。 その save() 末尾に「Forge に同期」を非同期で発火するフックを追加すれば、追加 UI なしで「常時同期」が実現できる。

具体的に同期発火する操作（既に save() を呼んでいる箇所）:

  • 変数の追加・編集・削除（保存ボタン押下時）
  • カテゴリ・サブカテゴリの追加・リネーム・削除
  • 変数のカテゴリ間移動（D&D）
  • カテゴリの並び替え（D&D）
  • Undo / Redo
  • Import / Merge Import 完了時

debounce (500ms) でまとめる: 連続編集で都度 HTTP を撃たない。 最後の編集から 500ms 静かになったら 1 回だけ POST。

Forge 未起動時の挙動

同期失敗時のフロー state.save() ↓ localStorage 書き込み (常に成功) ↓ queueSync() でデバウンス ↓ POST /sync_variables ↓ [失敗] ヘッダーに ⚠ 「未同期 N 件」バッジ表示 localStorage に "pending_sync: true" フラグ ↓ 次回 Forge オンライン検知時 (起動・URL 変更時) に再送

Manager 起動時の挙動（Forge 側変更があった場合の取り込み）

Manager ページ読み込み時に GET /variables で Forge 側の最新を取得し、Manager の localStorage と比較。 差分があれば「Forge 側で変更があります。取り込みますか？」ダイアログ。

使用感への影響:
  • + 編集が即 Forge に反映、追加クリック不要
  • + 「🚀 Forgeに送信」ボタンはビルダーテキスト送信のみに責任が縮小（変数定義は既に同期済み）
  • − Forge 未起動だと右上に「⚠ 未同期 N 件」バッジが常時表示される（うっとうしいと感じる可能性 → 設定で OFF にできるようにする）
  • − 編集後 500ms はネットワーク待ちが発生（無視できる程度）
リスクと回避:
  • 競合（Manager と Forge 側で同時編集）: Phase A では「Manager が常に正」(Manager の save が必ず勝つ)。Phase B で Forge 側に CRUD API を作る際に last-write-wins ロジックを導入
  • 大量編集で debounce が効かないケース: 500ms 後にまとめて 1 回だけ POST するため、典型的な編集パターンでは問題なし

2既存エントリ: 完全置換 + 双方向同期の道筋

回答 完全置換でいくが、Forge 側からも変数定義を追加できるよう改修したい。常に同期した状態を作る方針を提示してほしい。

✓ 決定: Phase A「Manager → Forge 完全置換 (片方向)」 → Phase B「Forge 側 CRUD API + Manager 取り込み (双方向)」

Phase A (今フェーズ) Manager → Forge 片方向、完全置換

Manager の state.save() → POST /sync_variables { mode: "replace", categories: [...] }。 variables.json 全体を Manager の現状で上書き。

初回同期前は vars/variables.json を vars/variables.json.bak.{timestamp} に自動退避（過去のローカル定義を失わない保険）。

Phase B (将来) Forge 側からも追加可能にする

「Forge 側で variables.json に追加 → Manager に反映」を成立させる経路:

案 B-1: Forge 側に CRUD UI を作らず、JSON 手動編集 + Manager 起動時取り込み

ユーザーがエディタで vars/variables.json を直接編集 → Manager ページ再読み込み時に「Forge 側が新しい」を検出し、差分を Manager に取り込む。

取り込みフロー Manager 起動 ↓ GET /variables ← variables.json を取得 ↓ Manager の localStorage と diff (両方が ETag/mtime を持つ) ↓ [Forge 側に新規/変更あり] ↓ ダイアログ:「Forge 側で 3 件追加されています。取り込みますか？」 ↓ Yes state.categories にマージ → state.save() → sync (循環防止のため取り込み時はフラグで sync 抑止)

案 B-2: Forge UI に簡易 CRUD ボタン

v3b の 📖 記法・使い方 に「変数を 1 件追加」「変数管理を開く」ボタンを追加。 押すと Manager ページを新規タブで開く or 簡易フォームを Forge 側に表示。 Phase B の中でもさらに先のフェーズ。

双方向同期のための競合解決

状況 解決方針
Manager 編集 中に Forge 側 が変わった save() 時に「Forge 側が新しい。上書き? 取り込み?」ダイアログ
同名変数が両側にある（content 違い） last-write-wins（updated_at を比較、後勝ち）
使用感への影響:
  • + Phase A 完了時点で「Manager 編集 → 即 Forge 反映」が常に成立
  • + Phase B 完了時点で「Forge 側手編集 → Manager に取り込み」も可能
  • − Phase A 中は Forge 側の手編集は次回 Manager save 時に消える（バックアップから復旧は可能）

3positive / negative フィールド分離 (schema v2)

回答 JSON でフィールドを分ければよい。Forge 側の変数読み込み処理を変える必要はあると認識。

✓ 決定: variables.json schema を v2 に拡張、positive/negative を分離して持つ

新スキーマ (v2)

vars/variables.json schema v2 { "schema_version": 2, "categories": [ { "name": "characters", "children": [ { "name": "pokemon", "variables": [ { "name": "ナンジャモ_ジム_雷_", "positive": "iono_(pokemon), pokemon, 1girl, pink_eyes", "negative": "", "updated_at": "2026-05-24T10:30:00Z" }, { "name": "カミツレ_ジム_電気_", "positive": "elesa_(pokemon), pokemon", "negative": "low quality, bad anatomy" } ] } ], "variables": [] } ] }

Forge expander 側の変更

variables_store.load_variables() を schema v2 対応に拡張:

load_variables の挙動 (改修後) def load_variables(path): data = json.load(...) if data["schema_version"] == 1: # 既存 v1 互換: values → positive のみ展開 return { "positive": flatten(data["variables"]), "negative": {}, } if data["schema_version"] == 2: # ツリーを walk して flat dict にする positive_dict = {} negative_dict = {} for var in walk_tree(data["categories"]): positive_dict[var["name"]] = [var.get("positive", "")] if var.get("negative"): negative_dict[var["name"]] = [var["negative"]] return { "positive": positive_dict, "negative": negative_dict }

expander.expand_prompts() はメインプロンプト展開で positive_dict のみ参照。 ネガティブプロンプト展開（将来 v3b で対応）は negative_dict を参照する設計。

$!foo 構文の扱い

Manager 側に既に $!foo 表記があるが、schema v2 では 1 変数に positive/negative 両方が入るので、$!foo という別変数概念は不要になる。 既存の $!foo エントリは migration で {name: "foo", negative: "..."} 形式の対応する変数に統合（または negative-only 変数として name そのままで保存）。

使用感への影響:
  • + 1 つの変数に positive と negative がペアで持てる（管理しやすい）
  • + ネガティブプロンプトへの流入経路を将来 v3b に組み込む布石になる
  • − schema v1 → v2 migration が必要（既存 variables.json は自動で .bak 退避してから v2 形式で再保存）
  • − 既存メインプロンプト中の $!foo 参照は手動修正必要（migration ログで一覧表示）

4カテゴリ階層: 単一ツリー JSON vs フォルダ分割複数 JSON

回答 Manager の階層を JSON 内のツリーで持つのではなく、vars/ フォルダ配下にカテゴリごとに JSON ファイルを分割する案ではどうか。両案を保守性の観点で比較してほしい（Forge 側読み取り改修も含めて）。

2 つの案の具体的なディスク構造

案 P (Plan): 単一ツリー JSON（v2 案） vars/ variables.json ← 1 ファイルにツリー全体 # 中身: { "schema_version": 2, "categories": [ {"name": "characters", "children": [ {"name": "pokemon", "variables": [{"name": "nanjamo", ...}]} ]}, {"name": "outfits", "variables": [...]} ] }
案 F (Folder): フォルダ分割複数 JSON（ご提案案） vars/ characters/ pokemon.json ← {"variables": [{"name": "nanjamo", ...}]} aether.json outfits/ school.json casual.json moods.json ← トップ直下に置く変数 # pokemon.json の中身（例）: { "schema_version": 2, "variables": [ {"name": "nanjamo", "positive": "iono_(pokemon)", "negative": ""} ] }

保守性 7 観点での比較

観点 案 P: 単一ツリー JSON 案 F: フォルダ分割複数 JSON
① 手編集のしやすさ 変数 100 件超になると 1 ファイルが長大、目的の変数を探しにくい ○ カテゴリ単位で小さなファイル、目的の変数にすぐ到達
② Git diff の可読性 どこか 1 変数を編集しても variables.json の差分に全て載る → ノイズ多い ○ 編集したカテゴリのファイルだけ変わる、レビューしやすい
③ Forge 側読み取り実装量 ○ json.load 1 回 + ツリー walk = ~30 行 os.walk(vars/) で再帰、各ファイル load、同名変数の重複検出、フォルダ名 = カテゴリ名 のマッピング = ~100 行
④ 同期 API 実装量 ○ POST /sync_variables で 1 ファイル全置換 + 自動 backup ＝ atomic 「どのファイルを書き換えるか」「不要になったファイルを削除」「フォルダ rename / 作成」を全部実装。atomic 化のために tmp ディレクトリ書き → リネームが必要

⑦ バックアップ / リストア ○ 1 ファイルを .bak.{ts} にコピーで完了 vars/ ディレクトリ全体を tar/zip で退避。リストア時に既存ファイルとの差分処理が要る

使用感（手編集視点）の比較

シナリオ 案 P での操作 案 F での操作
「pokemon カテゴリの 1 変数を直したい」 vscode で variables.json 開く → ファイル内検索 → 編集 vscode で vars/characters/pokemon.json を直接開く
「friend にカテゴリ単位で共有したい」 variables.json 内の該当カテゴリ部分を切り出して送付 pokemon.json を 1 つコピーして送付
「カテゴリの全変数を一括削除したい」 Manager で右クリック → 削除（自動同期） Manager で削除（同上）。手で消すなら pokemon.json を rm
「全変数を別マシンに移したい」 variables.json を 1 つコピー vars/ ディレクトリごとコピー

✓ 私の推奨: 案 P（単一ツリー JSON）

理由:

  1. 実装工数が約 1/3（300 行 → 100 行）。バグの混入リスクも低い
  2. atomic 操作が容易。1 ファイル書き換えなら部分失敗の心配なし。案 F は「フォルダ rename 中にクラッシュ → 中途半端な状態」というケースを考慮する必要あり
  3. 常時同期との相性が良い: Manager の state.save() 1 回 = ファイル 1 回書き換え。500ms debounce が単純に効く
  4. 手編集のしやすさという案 F の主要メリットは、変数管理 HTML を直接使えば不要（手編集する場面が少ない）

変数数が 1000 を超えるような規模になったら案 F に移行する余地は残しつつ、Phase A は案 P で開始。

案 P で進めた場合の使用感への影響:
  • + Manager で編集 → 500ms 後に variables.json 全置換 ＝ 単純で予測可能
  • + バックアップは variables.json.bak.{ts} 1 ファイルで完結
  • − 変数 100 件超で variables.json が「縦に長いファイル」になる（IDE のアウトライン機能でカテゴリにジャンプ可能）
  • − 友人にカテゴリ単位で共有する場合は手で切り出す必要あり（将来「カテゴリ Export」機能を Manager に追加する余地あり）
仮に案 F を選んだ場合のリスク:
  • フォルダ rename API のクロスプラットフォーム挙動（Windows のファイルロック / Linux の inode 挙動）
  • 同名変数が異なるカテゴリ内に存在する場合の競合検出ロジック
  • file watcher（将来 Forge 側で自動リロード機能を入れる場合）で大量のイベント発火
  • Atomic を保証するための tmp ディレクトリ → renamedir 手順の信頼性

＋変数名バリデーション: 入力時点で弾く

回答 使える文字列が異なるなら、そもそも入力できないようにしてほしい。

✓ 決定: Manager 側の入力フォームでリアルタイム入力制限 + Forge expander と同一文字種ルールに統一

許容文字の統一仕様

許容文字 備考
先頭 1 文字 非数字の word 文字（英字 / ひらがな / カタカナ / 漢字 / _） Forge expander の [^\W\d] と一致
2 文字目以降 word 文字 + ( ) : 、 Forge expander の [^=\s;,\$] と一致
禁止文字（入力時に弾く） 半角スペース / 改行 / = / ; / , / $ / / / { } / [ ] / . / | SD プロンプト構文の予約文字 + Forge 区切り文字

Manager 側 UI 実装

変数追加・編集モーダルの input <input type="text" id="varName" pattern="^[^\W\d][\w()：、:()]*$" oninput="validateVarName(this)" placeholder="$char, $ナンジャモ(ジム:雷) など"> function validateVarName(input) { const FORBIDDEN = /[/;{}\[\].,=|\s\$]/g; if (FORBIDDEN.test(input.value)) { input.setCustomValidity('使用できない文字: / ; { } [ ] . , = | スペース $'); input.reportValidity(); // 自動削除（ペースト時にも対応） input.value = input.value.replace(FORBIDDEN, ''); } else { input.setCustomValidity(''); } }

既存の自動置換 (_migrateVariableNamesV2) との関係

v1 から残っている _migrateVariableNamesV2（( : なども _ に置換していた）は過剰になる。 Forge 側が ( ) : 、 を許すなら、Manager 側も同じく許して置換ではなく許容に変更。 既存ユーザーの変換済みデータはそのまま保持し、新規入力では ( 等を許可。

使用感への影響:
  • + 入力時点で「使えない文字を打ったら入力されない」明確なフィードバック
  • + Manager で書ける名前 = Forge で参照できる名前、で完全一致
  • + $ナンジャモ(ジム:雷) のような表現力の高い名前が Manager でも使える
  • − 既存ユーザーの「v2 migration で _ 置換済み」変数は手動で名前を戻したい場合あり（オプション機能として「逆 migration ツール」を提供することも検討可）
