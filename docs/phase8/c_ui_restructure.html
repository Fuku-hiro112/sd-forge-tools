核心の設計判断

📂 途中再開 Accordion は Batch Resume Script に残置。理由: 履歴 v2 と再開ロジックは Script 選択時にのみ意味を持つ機能であり、AlwaysOn 化すると常時履歴ファイルを読みに行く負荷とトーストノイズが増えるため。

データ流の新しい姿

CASE AScript 未選択 (Forge 標準生成)

1. Click
User がメインプロンプトに $char, masterpiece + Generate
2. AlwaysOn process
prompt_expander.process(p, enabled, expansion_order_df, seed_mode, …) が呼ばれる
3. 展開
UI から取得した順序で expand_prompts → p.all_prompts / p.all_seeds / p.all_negative_prompts / p.n_iter を書換
4. 標準生成
process_images() が p.all_prompts を順次処理して N 枚生成

CASE BBatch Resume Script 選択 + 新規モード

1. Click
User が Batch Resume を選択して Generate (再開チェック OFF)
2. AlwaysOn process
prompt_expander.process() が動作 → p.all_prompts 等を書換。従来の _is_batch_resume_active ガードは撤去
3. Script run
batch_resume.run(p, history_dropdown, resume_check) が呼ばれる
4. 履歴記録
resume_check=False のため新規エントリを history_v2 に作成 (展開順は p から逆引き)
5. 生成ループ
batch_runner が p.all_prompts をイテレート + 各画像生成 + 進捗 update_entry。展開ロジック (expand_prompts) 呼出は撤去

CASE CBatch Resume Script 選択 + 再開モード

1. Click
User が履歴を選択 + 再開チェック ON + Generate
2. AlwaysOn process
prompt_expander が展開して p.all_prompts を書く (CASE B と同じ)
3. Script run
batch_resume が p.all_prompts を上書き: 履歴エントリの used_variables + expansion_order から再構築。同一順序での再開を保証
4. スキップ
history.progress.completed まで continue でスキップ、残りを生成

リスクと緩和策

Variable Manager JS の 🔄 / 📥 ボタン連携

「🔄 メインから取得」「📥 Variable Manager → メインに投入」は Manager 側の JS と elem_id で連携している。elem_id が変わると外部連携が壊れる。

elem_id を v3b 時代と同一に保つ。移管後に Manager JS 側の参照を一覧確認。
Forge AlwaysOn 拡張同士の処理順序

dynamic-prompts や ADetailer 等の AlwaysOn と sorting_priority が衝突する可能性。

prompt_expander.sorting_priority = 5 (低め) で展開を早期実施。L2 で ADetailer / Hires.fix / sd-forge-couple 併用確認 (R-10〜R-12)。
p.all_prompts 書換後の Forge 標準パスの挙動

p.n_iter / p.batch_size を書き換えるとプログレスバーや infotext の seed 表示が崩れる可能性。dynamic-prompts と同パターンだが Forge 固有差分は要確認。

UI-2 直後に手動で 1 回検証 (R-03)。崩れたら p.iteration / p.all_subseeds 等の追加調整。
再開モードでの整合性

Batch Resume が p.all_prompts を上書きするタイミング次第で、最初の 1 枚が prompt_expander 版で生成されてから batch_resume 版に切り替わる可能性。

batch_resume.run() の冒頭で resume 判定 → 即時 p.all_prompts 書換 → process_images 呼出、の順序を厳守。L2 で R-07 (Interrupt → 再開) を確認。
ロールバックコスト

UI-3 / UI-4 で複数ファイルが大きく変わるため、問題発覚時の戻し作業が大規模。

Step ごとに独立コミット (UI-1〜UI-4)。各コミットで動作テスト可能な状態を維持。問題は git reset --hard <tag> で即座に戻せる。
展開順 UI のセッション間永続化

v3b 時代は Script 選択するたびに UI が再構築されていた。AlwaysOn 化で Gradio セッション間で値が変わる挙動の差が出る可能性。

UI-1 完了後にブラウザリロード → 展開順 Dataframe の初期値が空であること確認。空 → 自動展開で同等動作。
Ⅴ.

合意ポイントのまとめ

  • ①展開順 / シードモード / 記法ヘルプ / 操作ボタンを Prompt Expander Accordion に移管
  • ②履歴 / 再開チェックは Batch Resume Script に残置 (Script 選択時のみ表示)
  • ③変数展開は 常に prompt_expander が担当。batch_resume は p.all_prompts を消費 (再開時のみ上書き)
  • ④UI-1〜UI-4 の 4 コミット分割で段階実装、各コミットでロールバック可能
  • ⑤L2 動作確認は UI-2 / UI-4 完了直後に簡易確認 (R-03 単発生成 / R-07 再開)
  • ⑥Variable Manager の JS 連携 (elem_id) は v3b 時代と同一に保つ
