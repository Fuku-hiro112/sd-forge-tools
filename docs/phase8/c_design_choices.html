SECTION VIII-C — PROMPT EXPANDER 分離

展開ロジックをどこに置くか。

Phase 8-C1 で prompt_expander.py の骨組は完成。次の C2〜C5 で「実展開ロジック」「展開順 UI」「API」「ネガティブ展開」をどう配分するかで、3 つの設計が存在する。

分岐点の核心

現状 (Phase 8-B 時点): 変数 $char の展開は v3b/batch_runner の中だけで起きる。Script 未選択の通常生成では展開されない。

Phase 8-C で目指す姿: prompt_expander が AlwaysOn で展開する ことで、Script 未選択でも $char が効くようになる。

▸ 展開ロジック本体の置き場所 — batch_runner (現状) / prompt_expander (移管) のどちら
▸ 展開順 UI (▲▼で並び替える Dataframe) の置き場所 — batch_resume Script / AlwaysOn 拡張
▸ API エンドポイント (variables/sync_variables/SSE) の登録元 — api_endpoints.py / prompt_expander.py
▸ ネガティブ自動展開の置き場所 — batch_runner / prompt_expander

本エージェントの推奨: Option A (保守的)

理由: L2 動作確認をユーザーがスキップしている現状で、Option B/C の広範な動作変更は気付けないリグレッションを生むリスクが高い。Phase 8-C は「prompt_expander という箱を作る」ことに集中し、ロジック移管は L2 を有人で実施できる段階で行うのが安全。

Option A でも「Script 未選択で $char が効くようになる」という Phase 8-C の主要価値は獲得できる。展開ロジックの重複は 8-D / 8-E 完了後に別途解消すれば良い。

注意: ただし「playbook 通りに進めたい」「気付けるリグレッションは自分で受け持つ」というスタンスなら Option B が標準案。本拡張の最終形に最短距離で到達できる。
